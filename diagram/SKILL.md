# Diagram — Codebase Architecture Visualizer

Generates a visual Mermaid flowchart of the project's architecture using lens for symbol-aware structural analysis. Outputs a self-contained `ARCHITECTURE.md` file at the project root.

## Tooling — lens mode vs fallback

Lens is the primary analysis engine. Fall back to filesystem tools only when lens isn't installed or indexes 0 symbols.

| Need | Lens mode | Fallback |
|---|---|---|
| Directory-level architecture | `lens map --depth 2` | `find` + `ls` |
| Entry points and top-level structure | `lens query "entry point main" --budget 1000` | `Grep` for `fn main` / `if __name__` / `app.listen` |
| Database / data store symbols | `lens query "database connection pool" --budget 1000` | `Grep` for connection strings, ORM init |
| Module A → Module B relationships | `lens path "symbolA" "symbolB"` | manual `Grep` for imports |
| What calls what within a module | `lens refs <symbol> --limit 15` | `Grep -rn <symbol>` |
| Symbol definition + callers | `lens follow <symbol> --budget 1000` | `Grep -n <symbol>` + `Read` |
| Cross-language disambiguation | `lens follow <symbol>` (auto-tags language) | manual file extension scan |

Prefer lens calls — they are budget-capped and symbol-aware, returning structural relationships without dumping whole files. Use `Read` only when lens can't answer the question (e.g. config file contents, non-indexed file types).

## The loop

### 1. Scan (lens-first)

1. **Bootstrap check.** Determine if lens is available:
   - `command -v lens` — if absent, set mode to fallback.
   - If present, check `.lens/index.db`. If missing or stale, run `lens init && lens index`.
   - If lens indexes 0 symbols, set mode to fallback.

2. **Architecture overview.** In lens mode:
   ```
   lens map --depth 2
   ```
   This returns a directory-level architecture summary with hot-spot ranking — the single most valuable command for understanding project structure without reading files.

3. **Entry-point discovery.** In lens mode:
   ```
   lens query "entry point main server startup" --budget 1000
   ```
   Returns symbol-graph seeds for the application's entry points with `file:line` anchors.

4. **Data-store discovery.** In lens mode:
   ```
   lens query "database connection pool cache queue storage" --budget 1000
   ```
   Locates data stores, caches, and message queues.

5. **Module relationship mapping.** For each top-level module identified in step 2, trace its connections:
   ```
   lens follow <key symbol in module> --budget 1000
   lens refs <key symbol> --limit 10
   ```
   Build a directed graph of which modules import/call which other modules.

6. **Cross-module paths.** For non-obvious relationships:
   ```
   lens path "<symbol in module A>" "<symbol in module B>"
   ```
   Reveals the call chain connecting distant modules.

### 2. Diagram

Generate a Mermaid `flowchart TB` (top-to-bottom architecture style):

- **Nodes** — top-level modules identified by lens. Label with real directory or crate names.
- **Edges** — directional arrows showing call/import relationships discovered by `lens refs` and `lens path`. Label edges with the relationship type where lens surfaces it (e.g. `gRPC`, `HTTP`, `event`).
- **Data stores** — nodes shaped `[( )]` (cylinder) for databases, caches, queues detected via `lens query`.
- **External services** — nodes shaped `[/ /]` (parallelogram) for external APIs and services.
- **Entry points** — nodes shaped `{{ }}` (hexagon) for `main`, `App::new`, server bootstrap symbols.
- **Subgraphs** — group related modules (`src/` core, `tests/`, `scripts/`, `config/`).

Constraints:
- Fit on one screen: 15–25 nodes, ≤5 subgraphs.
- Every node label comes from a lens result or a real directory name — never invent.
- No emojis, no HTML tags, no `\n` escapes in labels.
- Node IDs must be alphanumeric camelCase, no spaces or underscores.
- If the project has ≤3 source files, output a text tree instead of a Mermaid diagram.

### 3. Output

1. Write the diagram to `ARCHITECTURE.md` at the project root:
   ```markdown
   # Architecture — <project name>

   > Generated by `/diagram` on <ISO date>. Regenerate to refresh.

   ```mermaid
   flowchart TB
       ...
   ```
   ```

2. If `ARCHITECTURE.md` already exists, replace only the mermaid code block — preserve any user-written prose outside it.
3. Display a compact text summary: node count, key relationships found, lens commands used.

## Hard rules

- **Lens first.** Always start with `lens map`, `lens query`, and `lens follow`. Fall back to `find`/`Grep`/`Read` only when lens is absent or the index is empty.
- **Fresh scan every time.** Run lens commands in the current session — do not generate from memory, a prior diagram, or stale code-map notes.
- **Node labels come from lens output or real paths.** Never invent module names.
- Output one file: `ARCHITECTURE.md`. Do not scatter diagrams across the repo.
